These are stronger than generic token charts because they describe the shape and reliability of workflows, not just aggregate volume.
The key distinction is cohorting by session start date versus activity date. For session-level metrics, grouping by start date is usually correct: each session appears once, and we can ask whether sessions started that day behaved differently.
Cache Efficiency Distribution
A box-and-whisker plot per start date could show session-level cached-input share:
cache read / (uncached input + cache read + cache write)
Pros:
- Shows median and variability
- Reveals days where some sessions cached well and others failed badly
- Outliers identify particularly poor sessions
Caveats:
- Daily cohorts may have too few sessions for meaningful quartiles
- Sessions with only one call have no prior cache opportunity
- Models and harnesses have different cache semantics
Recommended behavior:
- Default to weekly buckets if there are fewer than roughly 10 comparable sessions per day
- Exclude baseline-only sessions from the distribution
- Show sample size in the tooltip
- Overlay or label the median
Session Duration Percentiles
Average, median, and p90 are useful, but plotting all three as independent lines could become noisy.
Recommended display:
- Median line
- Shaded p50–p90 range
- Average available in tooltip
- Session count shown per bucket
Duration definition:
last model/tool completion - first user turn
Caveats:
- Abandoned or still-active sessions can distort duration
- Long pauses between user turns may represent breaks, not agent runtime
- Some harness timestamps are less precise
Potentially split this later into:
- Wall-clock session span
- Active model/tool time
- Median turn duration
“Wall-clock span” is a safer label than “duration” if idle gaps are included.
Session Input Tokens
I agree that input is more useful than processed tokens for this analysis.
I’d define model input volume as:
uncached input + cache read + cache write
That represents all input context processed by the model across calls. It excludes output and reasoning.
Avoid labeling it simply “input tokens” without explanation because providers report cache reads and writes differently. A tooltip can state the exact formula.
Useful percentile view:
- Median model input volume
- p90
- Average in tooltip
- Optional toggle between per-session and per-turn
This helps identify context-heavy sessions even when caching keeps cost low.
Sessions Using Subagents
A simple trend works well:
sessions with ≥1 subagent / all sessions
Useful companion values:
- Median subagent count among sessions that use them
- Percentage of model calls occurring in subagents
- Percentage of cost occurring in subagents
A 100% stacked bar by start-date cohort could show:
No subagents | Used subagents
Subagent vs. Non-Subagent Cost
There are two different questions worth separating:
1. Cost composition
- Main-agent cost versus subagent cost within sessions that used subagents
- Best represented as stacked bars over time
2. Session comparison
- Cost distribution for sessions with versus without subagents
- Better represented as side-by-side box plots
The second can be misleading because difficult tasks are more likely to use subagents. It shows association, not “subagents caused higher cost.” Label it descriptively rather than as savings or overhead.
A stronger derived metric is:
subagent cost / total session cost
Show its median and p90 among sessions with subagents.
Additional Ideas
Cost Predictability
- Median and p90 session cost over time
- Percentage of sessions exceeding configurable thresholds such as $1, $5, or $10
- Useful for understanding risk, not only total spend
Expensive Session Concentration
- Percentage of spend generated by the top 10% of sessions
- Shows whether cost is broadly distributed or driven by a few outliers
Calls Per User Turn
- Median and p90 model calls generated per user turn
- Identifies agent loops and workflow amplification
- Split between main-agent and subagent calls
Context Growth
- Input volume by turn number
- Median context size for turn 1, turn 2, turn 3, etc.
- Shows how quickly conversations become expensive
This could be one of the most insightful charts in the product.
Cache Failure Position
- Percentage of misses by call position or turn number
- Answers whether cache failures happen near session start, after tools, after compaction, or late in long sessions
Cost Per Turn
- Median and p90 cost per user turn
- More comparable than total session cost when session lengths vary
Session Shape
Classify sessions into lightweight profiles:
- Single-turn
- Iterative conversation
- Tool-heavy
- Subagent-heavy
- Long-context
Then compare cost, duration, and cache outcomes across those groups.
Harness Reliability
For each harness:
- Full-miss session rate
- Median input volume
- Median duration
- Subagent usage rate
- Median cost
A compact comparison table with bars may work better than separate charts.
